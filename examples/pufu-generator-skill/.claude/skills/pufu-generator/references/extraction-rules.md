## 抽出の基本原則

1. **文脈を重視**: 単語のマッチングだけでなく、文脈から判断する
2. **候補を多めに抽出**: 最終的な選定は人間が行うため、多めに候補を出す
3. **出典を記録**: どのファイル・どの部分から抽出したかを記録する
4. **曖昧さを許容**: プロジェクト初期では曖昧な記述が多いことを理解する

## 獲得目標（gainingGoal）の抽出

**定義**: プロジェクトの目標・ミッション

**JSONキー**: `gainingGoal`

**キーワードパターン:**
```
# 高確度のキーワード
- ゴール、目標、目的、ミッション
- 成果物、アウトプット、deliverable
- 達成したい、実現したい、完成させる
- 〜の構築、〜の開発、〜のリリース

# 中確度のキーワード
- 目指す、ターゲット
- 最終的に、最終成果
- プロジェクトの目的
```

**抽出例:**
| 原文 | 抽出結果 | 確信度 |
|-----|---------|-------|
| "本プロジェクトのゴールは新規ECサイトの構築です" | 新規ECサイトの構築 | 高 |
| "最終的に社内システムの刷新を目指します" | 社内システムの刷新 | 高 |

## 勝利条件（winCondition）の抽出

**定義**: プロジェクト成功の判断基準・評価指標

**JSONキー**: `winCondition`

**キーワードパターン:**
```
# 高確度のキーワード
- 成功条件、達成条件、判断基準
- KPI、KGI、評価指標
- 〜を達成したら、〜になったら成功
- 合格ライン、クリア条件

# 中確度のキーワード
- 〜以上、〜以下、〜%
- 目標値、ターゲット数値
- ベンチマーク
```

**抽出例:**
| 原文 | 抽出結果 | 確信度 |
|-----|---------|-------|
| "月間売上1000万円を達成する" | 月間売上1000万円達成 | 高 |
| "ユーザー満足度80%以上" | ユーザー満足度80%以上 | 高 |

## 中間目的（purposes）の抽出

**定義**: 勝利条件達成のための「あるべき状態」

**JSONキー**: `purposes`（配列）

**キーワードパターン:**
```
# 高確度のキーワード
- 〜している状態、〜になっている
- 〜が実現している、〜が完了している
- 〜が整っている、〜が準備できている

# 中確度のキーワード
- 〜できるようになる
- 〜が機能している
- 〜が浸透している
```

**抽出例:**
| 原文 | 抽出結果 | 確信度 |
|-----|---------|-------|
| "ユーザーが迷わず操作できる状態" | ユーザーが迷わず操作できる状態 | 高 |
| "決済システムが安定稼働している" | 決済システムが安定稼働している | 高 |

**注意点:**
- 「する」ではなく「している」「なっている」という状態表現を探す
- 通常3つ程度にまとめる

## 施策（measures）の抽出と色分け

**定義**: 各中間目的を実現するための具体的行動

**JSONキー**: `measures`（各purposeの中にネスト）

**キーワードパターン:**
```
# 高確度のキーワード
- 〜する、〜を行う、〜を実施する
- タスク、アクション、TODO
- 〜を作成、〜を開発、〜を導入

# 中確度のキーワード
- 計画、予定
- ステップ、フェーズ
- 〜の検討、〜の調査
```

### 施策の色分けルール

| 色 | 意味 | 判定キーワード |
|----|------|--------------|
| `red` | 目標達成のために当然取り組むべき主なアクション | 必須、絶対、最優先、重要、核心、本質 |
| `white` | 標準的なアクション | （デフォルト） |
| `green` | 面倒でも、やらなければならないこと | 調整、折衝、交渉、承認、申請、手続き |
| `blue` | 後々発生するかもしれない問題への予防策 | リスク、予防、備え、対策、障害、保険 |
| `yellow` | 資源に余裕があればやりたいこと | 余裕があれば、できれば、追加、オプション |

**抽出例:**
| 原文 | 抽出結果 | 推定色 |
|-----|---------|-------|
| "UIプロトタイプを必ず作成する" | UIプロトタイプを作成する | red |
| "ユーザビリティテストを実施する" | ユーザビリティテストを実施する | white |
| "関係部署との調整会議を開催する" | 関係部署との調整会議を開催する | green |
| "障害発生時のロールバック手順を整備する" | ロールバック手順を整備する | blue |
| "余裕があれば多言語対応を追加する" | 多言語対応を追加する | yellow |

## 廟算八要素（elements）の抽出

### ひと（people）
```
JSONキー: elements.people
キーワード: チーム、メンバー、担当、役割、人員、スキル、経験、顧客、価値観
抽出例: "開発チーム5名（エンジニア3名、PM2名）"
```

### お金（money）
```
JSONキー: elements.money
キーワード: 予算、コスト、費用、資金、投資、利益、〜円、〜万
抽出例: "初期予算500万円、運用費月額50万円"
```

### 時間（time）
```
JSONキー: elements.time
キーワード: 期間、スケジュール、納期、期限、作業時間、〜月、〜週間
抽出例: "2025年1月〜3月の3ヶ月間"
```

### クオリティ（quality）
```
JSONキー: elements.quality
キーワード: 品質、水準、レベル、基準、満足感、仕上がり、技術
抽出例: "パフォーマンス重視、レスポンス2秒以内"
```

### 商流/座組（businessScheme）
```
JSONキー: elements.businessScheme
キーワード: 契約、取引、パートナー、委託、発注、ビジネスモデル、価値の流れ
抽出例: "自社開発、外部デザイン会社に一部委託"
```

### 環境（environment）
```
JSONキー: elements.environment
キーワード: 市場、業界、組織、社会情勢、規制、トレンド、国・地域
抽出例: "EC市場拡大中、個人情報保護規制強化"
```

### ライバル（rival）
```
JSONキー: elements.rival
キーワード: 競合、競争、他社、シェア
抽出例: "大手ECモール、同業他社のオンラインストア"
```

### 外敵（foreignEnemy）
```
JSONキー: elements.foreignEnemy
キーワード: リスク、課題、障害、問題、依存、制約、邪魔
抽出例: "レガシーシステムとの連携が必須、技術的負債"
```

## 局面分割の検出ルール

### 時系列区切りのキーワード

| キーワードパターン | 確信度 | 例 |
|-------------------|-------|-----|
| フェーズN、Phase N | 高 | 「フェーズ1：企画」「Phase 2: Development」 |
| 第N段階、第N期 | 高 | 「第1段階：調査」「第2期：実装」 |
| Q1/Q2/Q3/Q4 | 高 | 「Q1: 要件定義」 |
| ステップN | 中 | 「ステップ1」 |
| 前半/後半 | 中 | 「プロジェクト前半」 |
| 〜月、〜年度 | 中 | 「4月〜6月」 |
| スプリントN、イテレーションN | 中 | 「Sprint 1」 |
| マイルストーンN | 中 | 「マイルストーン1：α版リリース」 |

### 振り返り局面の検出キーワード

ドキュメント内に振り返りに関する記述がある場合、それを振り返り局面のソースとして使用する。

| キーワードパターン | 確信度 |
|-------------------|-------|
| 振り返り、ふりかえり、レトロスペクティブ | 高 |
| 反省、課題、学び、教訓 | 高 |
| 実績、結果、成果、達成度 | 中 |
| KPT、Keep/Problem/Try | 高 |
| 良かった点、改善点 | 中 |
| 次回への申し送り | 中 |

### 振り返り局面の要素抽出ルール

振り返り局面では、通常の要素抽出に加えて以下を抽出する：

| 抽出対象 | キーワード | マッピング先 |
|---------|-----------|-------------|
| 達成事項 | 達成、成功、完了、実現 | コメント（color: green） |
| 未達事項 | 未達、課題、遅延、未完了 | コメント（color: red） |
| 学び・教訓 | 学び、気づき、教訓、知見 | コメント（color: blue） |
| 施策の実績 | 実施済み、完了、中止、延期 | 施策の色の再評価 |

### 施策の色の再評価（振り返り局面）

計画局面の施策の色を、振り返り時の実績に基づいて再評価する：

| 実績の状態 | 振り返り局面での色 |
|-----------|-----------------|
| 計画通り実施・成功 | `red`（主要アクションとして確認） |
| 実施したが効果不十分 | `white`（標準に格下げ） |
| 面倒だったが完了した | `green`（面倒だが必要だったことを確認） |
| 予防策が役立った | `blue`（予防策の有効性確認） |
| 余裕がなく未実施 | `yellow`（次局面へ持ち越し） |
| 中止・不要になった | 施策自体を除外または取り消し線テキスト |

## ファイル形式別のヒント

### PowerPoint (.pptx)

| スライドの種類 | 抽出対象 |
|--------------|---------|
| タイトルスライド | プロジェクト名、獲得目標 |
| 目次スライド | 全体構成の把握 |
| 背景/現状スライド | 廟算八要素（environment, rival） |
| 目標/ゴールスライド | gainingGoal, winCondition |
| 計画/ロードマップスライド | measures, time |
| 体制図スライド | people, businessScheme |

### Excel (.xlsx)

| シートの種類 | 抽出対象 |
|------------|---------|
| 予算シート | money |
| スケジュールシート | time, measures |
| タスク一覧シート | measures |
| 体制表シート | people |
| KPI管理シート | winCondition |

### PDF (.pdf)

- テキスト抽出とテーブル抽出を両方実施
- ページ番号を記録して出典を明確に
- 図表は抽出が困難なため、周辺テキストから推測

### テキスト/マークダウン (.txt, .md)

- 見出し構造を活用: H1=プロジェクト名、H2=各セクション
- 箇条書き: 施策の候補
- 太字/強調: 重要なキーワード

### Word (.docx)

- 目次: 全体構造の把握
- 見出し: PowerPointと同様に分類
- 表: Excel同様に構造化情報
